# Security Policy

yohakuforce は Salesforce 組織のメタデータを扱う性質上、セキュリティを最重要視する。
脆弱性を発見した場合、以下の方針で対応する。

---

## サポート対象バージョン

OSS 公開後、以下のバージョン体系で脆弱性対応を行う (Phase 6 以降に発効)。

| バージョン | サポート状況 |
|---|---|
| 最新メジャー (例: 1.x) | ✅ 完全サポート |
| 一つ前のメジャー (例: 0.x) | ⚠ クリティカルのみ 6 か月 |
| それ以前 | ❌ サポート終了 |

Phase 1〜5 (内部検証期) は最新コミットのみ対象。

---

## 脆弱性報告フロー

### 公開前の OSS (現在)

- 開発者と直接やり取りする
- GitHub Issue は **使わない** (公開化のリスク)

### OSS 公開後 (Phase 7 以降)

1. **GitHub Security Advisories** から非公開で報告 (推奨)
2. もしくは GitHub の **Private vulnerability reporting** (Settings → Security → Private vulnerability reporting) を有効化済の場合はそちらへ送信
3. 上記 2 経路が利用できない場合は、リポジトリ管理者に直接連絡 (連絡先は OSS 公開時に Phase 7 で `MAINTAINERS.md` に記載予定)
3. 受領から **3 営業日以内** に一次返答
4. 重大度評価と修正方針を **14 日以内** に共有
5. 修正リリース後、**90 日経過 または パッチ公開済み** の段階で公開ディスクロージャ

ディスクロージャは **責任ある開示 (Coordinated Disclosure)** の慣行に従う。

---

## 重大度の基準 (CVSS 3.1 ベース)

| 重大度 | スコア | 対応期限 (パッチリリース) |
|---|---|---|
| Critical | 9.0+ | 7 日以内 |
| High | 7.0–8.9 | 30 日以内 |
| Medium | 4.0–6.9 | 90 日以内 |
| Low | 0.1–3.9 | 次回マイナーリリース |

## 想定する脅威モデル

本 OSS に特有のリスクと、対応する防御策を以下に列挙する (詳細は [`IMPLEMENTATION_GUIDE.md` § AIコストとセキュリティ方針](./IMPLEMENTATION_GUIDE.md#aiコストとセキュリティ方針))。

| 脅威 | 防御策 |
|---|---|
| 顧客メタデータが AI に送信される | `.yohaku/secrets-rules.yaml` によるマスキング (Phase 1) |
| `.yohaku/graph.sqlite` の盗難 | OS キーチェーン連携による暗号化オプション (Phase 6) |
| 認証情報の流出 | 本 OSS は org に直接接続しない (sf CLI / DX MCP に委譲) |
| 悪意あるテンプレート (`scaffold/`) 注入 | サプライチェーンセキュリティ (npm 署名、Dependabot, Phase 6) |
| プロンプトインジェクション | AI 出力のスキーマ検証 + `source` 列で人手介入領域を保護 |
| ハルシネーションによる誤情報伝播 | 知識グラフ経由の決定的処理 + 一致率 CI |

---

## 報告に含めてほしい情報

- 影響を受けるバージョン (またはコミット SHA)
- 再現手順 (最小限の例)
- 期待動作と実際の動作
- 影響範囲 (どのデータが漏洩・改ざんされうるか)
- 提案される修正方針 (任意)

---

## 機密情報の取り扱い

報告内容、再現用ダミーデータ、ログを受領した場合:

1. 第三者と共有しない
2. 修正完了後、ローカルから削除
3. ディスクロージャ前に報告者へ事前確認

---

## 範囲外

- 利用者プロジェクト固有の脆弱性 (本 OSS の責務外)
- Salesforce 公式ツール (sf CLI / DX MCP / Code Analyzer) の脆弱性 → Salesforce へ直接報告
- 依存ライブラリの脆弱性 → `npm audit` / Dependabot で対応、上流に報告
