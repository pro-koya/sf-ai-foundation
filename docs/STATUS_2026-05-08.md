## 2. プロジェクトのゴール (再確認)

### 最終目的 (CLAUDE.md より)

> Salesforce に携わる人々の **時間と余白** を生み出すため、属人化と運用負荷を AI で構造的に解消する OSS 基盤を作る。

### 段階的ゴール (README より)

| 段階 | ゴール | 現状 |
|---|---|---|
| **内部検証** | 自分の参画プロジェクトに適用し、本当に使えるかを実証する | **🟡 部分達成** — yohaku-trial (ダミープロジェクト) で全機能動作確認済み。実プロジェクト適用は未 |
| **社内展開** | 社内の他プロジェクトへ展開 | **⬜ 未着手** |
| **社外展開** | 個人 / 会社の支援事業として、コンサル導入や OSS 公開 | **⬜ 未着手** (公開前) |

### 解く中核課題 3 つ (README より)

1. **属人化の解消** — 仕様を知らない人が AI との対話だけで全体像を理解できる
2. **運用タスクの自動化** — ドキュメント整備 / 差分意味づけ / レビュー観点 / リリース資材 / 手動作業管理を AI が構造化
3. **AI 駆動開発の再現性** — ベンダーが現場間で知見を持ち運べる共通基盤と作法

## 3. 何が実装されているか

### 3.1 CLI コマンド (`yohaku`)

`packages/core/src/cli.ts` に登録済みのコマンドは以下:

| コマンド | 役割 | 状態 |
|---|---|---|
| `yohaku init [--bootstrap]` | 利用者プロジェクトに `scaffold/` を展開 | ✅ |
| `yohaku sync` | force-app/ → SQLite → 全 Markdown 描画を 1 コマンドで実行 | ✅ (主用途) |
| `yohaku graph build [--incremental] [--source local\|dx-mcp]` | 知識グラフ構築のみ | ✅ (dx-mcp は stub) |
| `yohaku graph query "<SQL>"` | SQLite に直接 SQL クエリ | ✅ |
| `yohaku graph schema [--format json\|markdown]` | グラフスキーマ出力 | ✅ |
| `yohaku render <target>` | 個別ターゲット (objects / flows / apex / triggers / permissions / validation-rules / approval-processes / sharing-rules / layouts / custom-metadata / named-credentials / remote-site-settings / lwc / aura / flexi-pages / system-index / system-overview / executive-summary / all) | ✅ |
| `yohaku diff` | Git 差分の決定的検出 + 7 カテゴリ分類 | ✅ |
| `yohaku validate` | 知識グラフ JSON Schema 検証 | ✅ |
| `yohaku metrics show / record` | AI コスト計測 (PostToolUse hook と連動) | ✅ |
| `yohaku onboard {context\|state\|faq}` | persona 別オンボーディング支援 | ✅ |
| `yohaku explain-write` | `/yohaku-explain` の AI_MANAGED 安全更新の書き込み口 | ✅ |
| `yohaku version` | バージョン表示 | ✅ |

### 3.2 取り込み済みメタデータ種別 (18)

| カテゴリ | 種別 | 出力先 |
|---|---|---|
| **オブジェクト系** (3) | SObject / Field / RecordType | `objects/` `record-types/` |
| **自動化系** (3) | Flow / ApexClass / ApexTrigger | `flows/` `apex/` `triggers/` |
| **権限系** (4) | PermissionSet / Profile / SharingRules / ValidationRule | `permissions/` `sharing-rules/` `validation-rules/` |
| **承認** (1) | ApprovalProcess | `approval-processes/` |
| **UI / レイアウト** (4) | Layout / LightningWebComponent (LWC) / AuraBundle / FlexiPage | `layouts/` `lwc/` `aura/` `flexi-pages/` |
| **構成** (3) | CustomMetadataRecord / NamedCredential / RemoteSiteSetting | `custom-metadata/` `named-credentials/` `remote-site-settings/` |

### 3.3 各 Markdown の構造 (3 層分離)

すべてのテンプレート (`packages/core/src/render/templates/*.eta`) は以下のブロック構造を持つ:

```
# <エンティティ名>

> Quick Summary (DETERMINISTIC、1〜3 行のスキャン用要約)

<!-- DETERMINISTIC_START id="..." -->
全件・件数・固定的な事実
<!-- DETERMINISTIC_END id="..." -->

<!-- AI_MANAGED_START id="purpose" -->
AI が業務的役割を推測した文 (`/yohaku-explain` で更新)
<!-- AI_MANAGED_END id="purpose" -->

<!-- HUMAN_MANAGED_START id="business-context" -->
人間が記述する顧客固有ルール・業務文脈 (AI は触らない)
<!-- HUMAN_MANAGED_END id="business-context" -->
```

再描画時、HUMAN_MANAGED は必ず保全。AI_MANAGED は `customized=true` フラグ付きなら保全 (Phase 8)。

### 3.4 主要描画機能

- **Quick Summary**: 全エンティティに 1〜3 行のスキャン用要約 (Phase 9-A2)
- **Mermaid 図**:
  - Trigger 起点の `flowchart LR` (呼び出し先クラス + SOQL/DML 対象)
  - Flow 全体の `flowchart` (要素 → 要素の遷移)
  - Apex メソッド単位の制御フロー
  - System Overview (Trigger / Apex / Flow / Object の俯瞰)
  - 承認プロセスの `flowchart`
- **自動懸念検出**: `concernsForApex/Trigger/Flow` で HIGH / MEDIUM / INFO の 3 段階ヒューリスティック (Phase 7-D)
- **上層部資料**: `executive/summary.md` (構築規模 + 品質指標) / `executive/risks.md` (全件懸念集約)
- **権限マトリクス**: PermissionSet / Profile の object × CRUD / field × R/W / class アクセス

### 3.5 Claude Code 統合 (scaffold 配布物)

`scaffold/.claude/` に以下が配布される:

## 4. 実行方法

### 4.1 開発側 (本リポジトリ)

```bash
# 依存解決 (workspace モノレポ)
npm install

# ビルド (core: tsc + アセットコピー)
npm run build

# 検証
npm run typecheck
npm test               # 226 件 pass を期待
npm run lint
```

### 4.2 利用者側 (Salesforce DX プロジェクト)

```bash
# 初回セットアップ (scaffold 展開 + force-app 読み込み + 全 Markdown 描画)
yohaku init --bootstrap

# 日常運用 (force-app/ の変更を Markdown に同期)
yohaku sync

# 個別描画
yohaku render apex
yohaku render flows
yohaku render lwc

# 差分意味づけ
yohaku diff
```

### 4.3 yohaku-trial での実機検証コマンド

`~/Desktop/yohaku-trial` (検証用ダミー DX プロジェクト) で:

```bash
node ~/Desktop/yohakuforce/packages/core/dist/cli.js sync
# 期待: [yohaku] render complete: written=66 archived=0 warnings=0
```

### 6.2 重要な乖離: 元計画 Phase 7 (普及) は未着手

**元計画**: Phase 7 = OSS 公開 / Issue 対応 / 月次アクティブ導入 10 件 / 公開導入事例 3 件 / メタデータカバレッジ 80%

**実際**: Phase 7 以降は **メタデータ取り込みカバレッジの拡充** に注力 (15 種 → 18 種)。

**理由 (推測 + ナレッジから)**:
- yohaku-trial で全機能を動かすと「描画されない種類のメタデータ」が目立ち、利用者体験を上げるには取り込み拡充が先と判断
- 元計画の Phase 7 はコミュニティ運営コスト・互換性維持コストの懸念があり、まず内部検証品質を上げる方を優先

**結果**:
- Markdown 化品質は当初想定を **超えている** (Mermaid / Quick Summary / 上層部資料 / 自動懸念検出)
- 一方で「実プロジェクトに 1 件適用」「OSS 公開」「導入事例」は **未達**

### 6.3 KPI 観点でのゴール乖離

| KPI (元計画) | 目標 | 現状 |
|---|---|---|
| 月次アクティブ導入プロジェクト | 6 か月で 10 / 12 か月で 30 | **0** (yohaku-trial = 検証用ダミーのみ) |
| GitHub Issue 中央値解決時間 | 14 日以内 | **N/A** (未公開) |
| 公開導入事例 | 12 か月で最低 3 件 | **0** |
| AI 出力一致率 | 95% 以上維持 | **枠組みのみ** (実 AI 統合は別タスク) |
| メタデータ型カバレッジ | 公開時 80% / 12 か月で 95% | **18 種 / 主要 ~25 種で 72%** (推定) |

### 6.4 内部検証 (= yohaku-trial) との関係

yohaku-trial は **本物の顧客プロジェクトではない**。`force-app/` のサイズは小さく、エッジケース (5000+ オブジェクト、複雑な権限階層、Aura → LWC 移行中の混在) をカバーしていない。

**ゴール (内部検証)** = 「現在参画中のプロジェクトに適用」だが、現在この適用は実施されていない。

## 8. 次アクションの選択肢

> ここは判断が必要な領域。以下に **A〜D の路線** を提示。それぞれ独立に進められるが、複数を同時並行するとレビュー負荷が増える。

### 路線 A: 実プロジェクト適用 (内部検証ゴールの真の達成)

**狙い**: 「自分が参画中の本物のプロジェクトに yohaku を入れて、本当に価値があるか実証する」

**やること例**:
1. 実プロジェクトの `force-app/` で `yohaku sync` を流す (失敗パターンを 1 サイクル分集める)
2. 出てきた 200+ ファイルの Markdown を **1 つだけ** 選んで、3 ヶ月後の自分が読んで業務理解できるか自己評価
3. 実プロジェクト固有の取りこぼしメタデータ種別 / エラーパターンを `pitfalls/` に蓄積
4. 顧客固有情報を含む可能性があるため、**メタ層 / 配布物層 / 顧客層** の物理分離を強化 (ADR 化)

**得られるもの**: 実プロジェクトでの不具合 / 不足機能の特定。Phase 12 以降の本物の優先順位。

**リスク**: 機密情報の混入、既存運用への影響、本業時間の圧迫。

### 路線 B: Visualforce / Lightning App / その他取り込み拡充 (Phase 12 系)

**狙い**: メタデータ型カバレッジ 95% へ近づける

**やること例**:
1. Visualforce (Page / Component) の取り込み
2. Lightning App (`*.app-meta.xml`) の取り込み
3. StaticResource / Tab / Application 系
4. 横断ドキュメント (権限マトリクス / 自動化マトリクス / ER 図)

**得られるもの**: より「死角のない」設計書化。

**リスク**: Phase 7 から続く「取り込み拡充ループ」に閉じ続けるリスク。利用者検証が伴わなければ価値が見えない。

### 路線 C: 元計画 Phase 7 (普及) に着手

**狙い**: GitHub 公開 / Issue テンプレ / 導入ガイド / コミュニティ運営の準備

**やること例**:
1. GitHub Issue / PR テンプレ
2. `docs/01-getting-started/` を本気で書く (利用者の最初の 5 分)
3. 公開前のセキュリティレビュー (顧客固有情報の混入チェック)
4. `examples/sample-project/` で「3 分で動く」サンプル整備
5. ライセンス確定 (Apache 2.0 で確定済) と NOTICE 更新

**得られるもの**: 公開可能な状態。

**リスク**: 内部検証 (路線 A) を経ずに公開すると「プロモ先行で実態が薄い」と見なされる。

### 路線 D: 再現性 / 一致率 CI の本気実装

**狙い**: AI 出力一致率 95% 維持 (元計画 Phase 7 の KPI) を技術的に保証する

**やること例**:
1. `/yohaku-explain` を温度 0 / プロンプトハッシュ / N-run 一致で固定する仕組み
2. Phase 8 の安全更新経路を CI に組み込み、回帰検出
3. ゴールデンテストを AI_MANAGED ブロックにも拡張

**得られるもの**: 「AI が出した内容が運用で揺れない」という再現性ガバナンス。

**リスク**: AI コストと CI 時間。

### 推奨

**まずは路線 A の最小版** をやって「実プロジェクトで何が足りないか」を 1 サイクル集めてから、路線 B / C / D を選ぶ判断軸を作るのが妥当に見える。

ただし「実プロジェクト」が今は無いなら、**路線 C の最小版** (`docs/01-getting-started/` の整備 + Issue テンプレ) を先に進めて公開ハードルを下げ、外部から実プロジェクト適用の機会を呼び込む手もある。
