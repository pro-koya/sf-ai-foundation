# Phase 12 着手計画

## 背景

Phase 11 までで 18 種のメタデータを取り込み、`yohaku sync` で 66 ファイルを再生成できる状態に到達。一方で利用者目線の指摘:

> マークダウン化された設計書がメタ情報によりすぎていて、設計書とは言えない。
> 何かはわかるが、何をしていて何のためにあるのかといったことがわからない。

この指摘は妥当。現状のテンプレートは DETERMINISTIC ブロック (件数 / リスト / 表) が主で、AI_MANAGED は各テンプレートに `purpose` の 1 ブロックしかなく、自然言語の余白が小さい。

加えて、Claude Code / Antigravity 前提なので、テンプレート枠は固定したまま **自然言語ブロックを増やす** 余裕がある。

## 判断

Phase 12 を 2 つの軸で進める:

| 軸 | 路線 | やること |
|---|---|---|
| **品質軸** | D | 設計書としての本物化 (AI_MANAGED ブロック拡充) |
| **網羅軸** | B | 取り込み拡充 (Visualforce / Lightning App) + ER 図 |

## 12-D: 設計書としての本物化

### 設計原則

1. **DETERMINISTIC は削らない** — 再現性の根なので、ファクト表現は維持
2. **AI_MANAGED ブロックを増やす** — 自然言語の余白を作る
3. **ブロック ID で意図を固定** — 内部は自由文だが、何を書く欄かは明示
4. **HUMAN_MANAGED は 1 つだけ** — 顧客固有文脈は分割しない (人間の負担を増やさない)

### 標準ブロック (新規追加候補)

| ブロック ID | 何を書くか | 対象エンティティ |
|---|---|---|
| `narrative` | 「これは何で、何のためにあるのか」を 2〜3 段落の自然言語で | **全エンティティ必須** |
| `business-scenario` | 業務シナリオ (Given/When/Then や具体例) | Apex / Flow / Trigger |
| `operational-notes` | 運用上の注意 (例外パターン / 依存関係 / 罠) | Trigger / Flow / Object |
| `business-domain` | どの業務領域 (請求 / 在庫 / 与信 等) に属するか | Object |
| `key-design-decisions` | 設計判断 (なぜこの構造か、何を捨てたか) | Apex |

既存の `purpose` ブロックは:
- LWC / Aura / FlexiPage / その他は **`purpose` のまま維持** (UI 系は narrative の必要性が低い)
- Apex / Flow / Trigger / Object は **`narrative` に置き換え + 追加ブロック**

### テンプレート別の改修方針 (重要: purpose は削除しない)

| テンプレート | 現状 AI_MANAGED | 改修後 AI_MANAGED (新規追加) |
|---|---|---|
| `apex-class.eta` | purpose / concerns | + **narrative** / **business-scenario** / **key-design-decisions** |
| `apex-trigger.eta` | purpose | + **narrative** / **operational-notes** |
| `flow.eta` | purpose | + **narrative** / **business-scenario** / **operational-notes** |
| `object.eta` | purpose | + **narrative** / **business-domain** |
| その他 (LWC / Aura / FlexiPage / Layout / PS / Profile / VR / RT / AP / SR / CMR / NC / RSS) | purpose | **改修なし** |

### 後方互換性 (重要)

- `purpose` ブロックは **絶対に削除しない**。既存ファイルの `/yohaku-explain` 出力 (例: `AccountBalanceService.md` の AI 推定文) を保全する必要がある (Phase 8 完了 ADR の保全要件)
- 新ブロック (`narrative` 等) は **追加** のみ。既存 Markdown の merge では「テンプレートに新規 ID が増えた」状態として扱われ、初回 sync 時にプレースホルダ文で埋まる
- `customized=true` フラグの保全は ID 別に独立しているので、`narrative` を AI で書いても `purpose` の既存内容は触らない

## 12-B: 取り込み拡充

### 12-B1: Visualforce Page (`*.page-meta.xml`)

- 起点: `pages/<Name>.page-meta.xml`
- 兄弟: `<Name>.page` (markup)
- 抽出: `controller` / `extensions[]` / `standardController` / `renderAs` (`pdf` 等) / `apiVersion` / `label` / `description` / `availableInTouch` / `confirmationTokenRequired`
- markup 解析 (regex): `<apex:repeat>` / `<apex:outputText>` / `<apex:form>` / `<apex:inputField>` / `<apex:commandButton>` / `<apex:repeat>` の件数

### 12-B2: Visualforce Component (`*.component-meta.xml`)

- 起点: `components/<Name>.component-meta.xml`
- 兄弟: `<Name>.component` (markup)
- 抽出: `controller` / `description` / `apiVersion` / `<apex:attribute name=...>` 一覧

### 12-B3: Lightning App / CustomApplication (`*.app-meta.xml` in `applications/`)

- ⚠ AuraBundle の `.app-meta.xml` (in `aura/<Name>/`) と区別が必要
- adapter classifier はディレクトリベースで分岐:
  - `aura/<Name>/<Name>.app-meta.xml` → `AuraBundle`
  - `applications/<Name>.app-meta.xml` → `CustomApplication` (新規)
- 抽出: `label` / `description` / `navType` / `tabs[]` / `utilityBar` / `formFactors[]` / `brand`

### 12-B4: ER 図 (Object 関係性)

- `graph.fields` の `referenceTo` を集約し、Master-Detail / Lookup を区別して Mermaid `erDiagram` 化
- 出力先: `system-overview.md` のセクション末尾 (新規 file は作らない、上層に集約)
