# Phase 5 着手計画

## 判断

Phase 5 を **6 サイクル** に分割。プロジェクト仕様を知らない人が **AI 対話のみで主要ドメインを理解できる** 状態を完成させる。本 OSS の「属人化解消」中核目的をここで実現する。

## サイクル構成

| # | サイクル名 | 主要成果物 | 完了条件 |
|---|---|---|---|
| 5-1 | context-map + persona 別読み順 | `.sfai/context-map.yaml` ローダ、4 persona の読み順テンプレ、scaffold テンプレ | YAML から persona 別に read_order が取得できる |
| 5-2 | onboarding-state + 進捗記録 | `.sfai/onboarding-state.json` read/write API、step 完了記録 | JSON で参画者の閲覧履歴が永続化される |
| 5-3 | persona 別 subagent (3 種) | scaffold の review-assistant / release-advisor / customer-impact-explainer | 各 subagent が責務を明示、tools 制限済み |
| 5-4 | `sfai onboard` CLI + `/onboard --role` 拡張 | コンテキスト表示・状態管理 CLI、slash command の role 分岐強化 | 4 persona で挙動が変わる |
| 5-5 | FAQ 蓄積機構 + PII フィルタ | 対話ログから頻出質問を抽出、secrets/mask 再利用で PII マスキング | docs/human/faq/ に AI_MANAGED + HUMAN_MANAGED で蓄積 |
| 5-6 | **検証ゲート + 振り返り** | sfai-trial で 4 persona × `/onboard --role` を実機検証 | retrospective、Phase 5 完了 ADR |

各サイクル末に retrospective 必須。Phase 5 全体: **2 週間相当**。

## 検証ゲート設計

### ゴール

利用者の sfai-trial で **4 つの persona × `/onboard --role <persona>`** を実行し、各 persona に応じた読み順・subagent 起動・進捗記録の挙動を確認。

### 計測項目

| 指標 | 単位 | 目標 |
|---|---|---|
| `/onboard --role new_joiner` 完走時間 | 分 | 30 分以内 (主要オブジェクト 5 件理解まで) |
| `/onboard --role reviewer` で直近 PR 把握 | 分 | 5 分以内 |
| `/onboard --role release-manager` で直近リリース把握 | 分 | 5 分以内 |
| `/onboard --role customer-facing` で顧客影響整理 | 分 | 10 分以内 |
| AI 任せられたタスク比率 | % | 60% 以上 |
| FAQ 候補抽出件数 | 件 | 検証セッション中に最低 3 件 |
| AI コスト | USD/セッション | 0.30 USD 以下 |

### 検証プロセス

1. sfai-trial で `sfai init --bootstrap --profile full --conflict overwrite` で再展開
2. `.sfai/context-map.yaml` を sfai-trial の実際のドメインに合わせて編集 (sales / finance 等)
3. Claude Code から `/onboard --role new_joiner` 実行 → 段階的提示確認
4. `/onboard --role reviewer`, `/onboard --role release-manager`, `/onboard --role customer-facing` を順次実行
5. `.sfai/onboarding-state.json` に進捗が記録されることを確認
6. FAQ 抽出を `sfai onboard faq extract` で試行
7. 結果を retrospective に記録

## 主要設計判断

### context-map.yaml の最小構造

```yaml
project:
  name: <projectName>
  domains:
    - id: sales
      description: 営業関連 (商談・受注)
      primary_objects: [Account, Order__c]
      key_docs:
        - docs/generated/objects/Account.md
        - docs/human/business-notes/sales-process.md
    - id: finance
      description: 経理・与信
      primary_objects: [Account, Invoice__c, Claim__c]
      key_docs: [docs/generated/objects/Account.md]

personas:
  new_joiner:
    goal: 2 週間で主要ドメインを理解する
    read_order:
      - docs/generated/system-index.md
      - docs/human/business-notes/overview.md
      - "domains:*"           # 全ドメインを順次
    depth: summary_first
  reviewer:
    goal: PR を適切にレビューする
    read_order:
      - docs/ai-augmented/change-summaries/latest.md
      - ops/registry/manual-steps-registry.md
    depth: detail_on_demand
  release-manager:
    goal: リリース準備の抜け漏れを防ぐ
    read_order:
      - docs/releases/latest.md
      - ops/registry/manual-steps-registry.md
    depth: checklist_first
  customer-facing:
    goal: 顧客視点で影響を整理
    read_order:
      - docs/ai-augmented/change-summaries/latest.md
    depth: business_impact_first
```

### FAQ 蓄積 + PII フィルタ

- 対話ログ (Markdown 形式 or Claude Code transcript JSON) を入力
- 頻出質問パターンを抽出 (簡易: 質問プレフィックス `?` または `〜って何` 等)
- 既存 `secrets/mask.ts` を **そのまま再利用** して PII マスキング
- 出力: `docs/human/faq/<topic>.md` に
  - `AI_MANAGED` ブロック: 抽出された質問・回答ペア
  - `HUMAN_MANAGED` ブロック: 人手精査領域 (AI 上書き禁止)

### persona 別 subagent

| persona | 既存 / 新規 subagent | 主な責務 |
|---|---|---|
| new_joiner | onboarding-guide (既存) | 段階的提示、対話的深掘り |
| reviewer | review-assistant (新規) | 直近 change_summary をレビュー観点で要約 |
| release-manager | release-advisor (新規) | 直近 release_doc + 抜け漏れ pitfalls 検出 |
| customer-facing | customer-impact-explainer (新規) | 顧客視点に翻訳、技術用語を業務用語に変換 |

## 代替案

| 案 | 採否 | 理由 |
|---|---|---|
| A. **6 サイクル分割 (本案)** | 採用 | Phase 3-4 と同流儀、各サイクル独立検証可能 |
| B. context-map を JSON にする | 却下 | YAML の方が人間が編集しやすい (利用者が頻繁に編集する想定) |
| C. onboarding-state を SQLite に格納 | 却下 | JSON 1 ファイルで十分、可読性優先 |
| D. FAQ を AI 単独で完全自動生成 | 却下 | ハルシネーションリスク。HUMAN_MANAGED で人手精査前提 |

## トレードオフ

- **代償**:
  - context-map.yaml を利用者が編集する手間 → サンプルテンプレートで初期負荷軽減
  - onboarding-state は gitignore のためチーム間で共有されない → Phase 7 でチーム共有版検討
- **将来課題**:
  - 多言語対応 (i18n) は Phase 7
