# Phase 14 着手計画

## 背景

Phase 12〜13 で AI_MANAGED ブロックを大量に追加した:

| エンティティ | 追加ブロック (Phase 12 + 13) |
|---|---|
| Apex Class | narrative / business-scenario / key-design-decisions / processing-overview-narrative / processing-details-narrative |
| Apex Trigger | narrative / operational-notes / processing-overview-narrative / processing-details-narrative |
| Flow | narrative / business-scenario / operational-notes / processing-overview-narrative / processing-details-narrative |
| Object | narrative / business-domain |
| LWC / Aura / FlexiPage / Visualforce / Lightning App | (purpose / narrative などエンティティ別に追加) |

しかし `yohaku explain-write` (Phase 8) は **Apex Class / Trigger / Flow の 3 種類のみ対応** で、Object / LWC / Visualforce / Lightning App 等の新エンティティはサポート外。また、各エンティティごとに「どのブロック ID に何を書くべきか」の指針が AI 側に明示されていない。

結果として:
- 新ブロック (`narrative` 等) は **テンプレートのプレースホルダ** のままで、`/yohaku-explain` ワークフローからは書き換えられない
- Object に `narrative` を書きたい場合の経路が無い
- Slash command / subagent が新 ID を知らないため、AI が生成すべき内容の指針も無い

## 判断

Phase 14 を **「AI_MANAGED ブロック自動生成のフルパス整備」** として実施する:

| 軸 | やること |
|---|---|
| **14-A** | `applyExplain` / CLI `explain-write` を **全エンティティ種別** に拡張 (Object / LWC / Aura / FlexiPage / Visualforce / Lightning App を追加) |
| **14-B** | **Block ID Registry** を新設し、各エンティティ種別が許容する AI_MANAGED ブロック ID をホワイトリスト化 |
| **14-C** | scaffold に `/yohaku-explain` slash command と `explain-writer` subagent を新設 |
| **14-D** | yohaku-trial で **3 エンティティ × 全 AI_MANAGED ブロック** を実機書き込み + 再 sync 保全検証 |

## 14-A: ExplainKind 拡張

現状の `ExplainKind` は `"apexClass" | "apexTrigger" | "flow"`。これを以下に拡張:

```typescript

`KIND_TO_SUBDIR` も同様に拡張。テスト追加。

## 14-C: Slash command + Subagent

### `/yohaku-explain` slash command

新規 `scaffold/.claude/commands/yohaku-explain.md.eta`:

```markdown
---
description: <%= it.projectName %> の AI_MANAGED ブロックを再生成する
argument-hint: <kind> <fqn> [block-ids...]
---

# /yohaku-explain $ARGUMENTS

知識グラフを参照して、対象エンティティの AI_MANAGED ブロックを再生成し、`yohaku explain-write` で書き戻す。

## 実行手順

1. 引数を分解:
   - `<kind>` = apexClass | apexTrigger | flow | object | lwc | auraBundle | flexiPage | visualforcePage | visualforceComponent | customApplication
   - `<fqn>` = エンティティ完全名
   - `[block-ids...]` = 対象ブロック ID (省略時は全 ID)

2. `yohaku graph query` で対象エンティティのファクトを取得 (DETERMINISTIC 情報)

3. **explain-writer subagent を起動** し、各 block-id に対する文面を生成させる

4. 生成結果を `/tmp/yohaku-explain-<fqn>.json` に書き出し:
   ```json
   {
     "narrative": "このクラスは ...",
     "business-scenario": "..."
   }
   ```

5. `yohaku explain-write --kind <kind> --fqn <fqn> --input /tmp/yohaku-explain-<fqn>.json`

## 禁則

- DETERMINISTIC ブロック (件数 / 表 / Mermaid) は AI 生成の対象にしない
- HUMAN_MANAGED ブロックには絶対書き込まない
- 既存の AI_MANAGED 内容を読まず、ファクトのみから生成する (再生成性を担保)
```

### `explain-writer` subagent

新規 `scaffold/.claude/agents/explain-writer.md.eta`:

各ブロック ID に対する **書くべき内容の指針** を明示:

| ブロック ID | 何を書くか |
|---|---|
| `purpose` | 1〜3 行の役割説明 |
| `narrative` | 「これは何で、なぜ存在するか」を 2〜3 段落 |
| `business-scenario` | 業務シナリオの具体例 (Given/When/Then) |
| `operational-notes` | 運用上の罠 / bulk 安全性 / 順序依存 |
| `business-domain` | 属する業務ドメインと隣接オブジェクト |
| `key-design-decisions` | 設計判断 (なぜこの構造か) |
| `processing-overview-narrative` | 処理の流れの俯瞰 |
| `processing-details-narrative` | メソッド/要素単位の処理詳細 |
| `concerns` | 自動検出に含まれない懸念 |

## 14-D: yohaku-trial 実機検証

擬似的な (= 手書きで作った) JSON 入力を `yohaku explain-write` に流し、以下を検証:

| 検証項目 | 期待結果 |
|---|---|
| Apex Class (`AccountBalanceService`) に 7 ブロック書き込み | 全 ID が `updated` で報告される |
| Object (`Order__c`) に 3 ブロック書き込み (新 kind) | エラーなく書き込み成功 |
| Flow (`Order_AutoCreateShipmentOnApproval`) に 7 ブロック書き込み | 全 ID 反映 |
| 不正な kind / 不正な block-id を渡した場合 | 事前検証でエラー表示 |
| 書き込み後に `yohaku sync` を再実行 | AI_MANAGED 内容が **完全保全** される (Phase 8 の保全契約を継承) |
| Apex の `purpose` (Phase 8 で書かれた既存内容) | 触らない場合は維持される |
