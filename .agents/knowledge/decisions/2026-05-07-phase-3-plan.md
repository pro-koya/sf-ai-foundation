# Phase 3 着手計画

## 判断

Phase 3 を以下の **8 サイクル** に分割して実行する。差分意味づけは Phase 1〜2 と異なり **AI 出力の再現性ガバナンスが本格稼働** する初の Phase であるため、ガバナンス層の構築 → 並列分類器 → 統合 → 検証ゲート の順で進める。

**前提**: Phase 2.5 (UX 改善サブサイクル) を Phase 3 着手前に完了させる。

## サイクル構成

| # | サイクル名 | 主要成果物 | 完了条件 |
|---|---|---|---|
| 3-0 | **Phase 2.5: UX 改善** | `yohaku init --bootstrap` / `yohaku sync` / `yohaku render` 引数省略 | 利用者が 1 コマンドで初回 / 日常運用ができる |
| 3-1 | 差分検出基盤 | `yohaku diff --from <ref> --to <ref> --json` | Git 2 リビジョン間の **決定的差分** を JSON 化 (構造のみ、意味づけ無し) |
| 3-2 | change_summary スキーマ + source 列 | JSON Schema、`Tracked<T>` ヘルパー、ajv で source 必須化 | スキーマ ADR、テストで source 未指定が弾かれる |
| 3-3 | 並列分類 subagent (5 種) | data-model / automation / permission / ui / logic の各分類器を `scaffold/.claude/agents/` に追加 | 各 subagent の system prompt と tools 制限が明文化 |
| 3-4 | `/classify-diff` slash command | Task ツールで 5 並列起動、結果統合 | scaffold に追加、profile 別有効化 |
| 3-5 | 一致率 CI 基盤 | temperature=0 で N 回再実行、80% 以上を CI で強制 | `tests/golden/ai/` に分類サンプル 3 件、一致率テスト |
| 3-6 | Code Analyzer SARIF 取り込み | `yohaku diff --include-static-analysis <sarif>` で SARIF を入力に | findings が change_summary に含まれる |
| 3-7 | `/change-summary` (PR サマリ) + 大型差分サンプリング | docs/ai-augmented/change-summaries/ への自動配置、1000 ファイル超は LIMIT | サンプル PR で動作 |
| 3-8 | **検証ゲート + 振り返り** | 1 つの実 PR で `/classify-diff` を実運用 | 分類精度を人手照合、retrospective 記録 |

各サイクル末に retrospective を必須。Phase 3 全体の期間目安: **2〜3 週間**。

## 検証ゲート設計

### ゴール

1 名が **1 つの実 PR** に対して `/classify-diff` を実行し、分類結果を人手照合する。

### 計測項目

| 指標 | 単位 | 目標 |
|---|---|---|
| `/classify-diff` 完走時間 | 秒 | 60 秒以内 (5 並列で大型差分でも) |
| 分類精度 (人手照合) | 完全一致率 | 80% 以上 |
| 一致率 CI スコア | % | 80% 以上 |
| 手動作業見逃し件数 | 件 | 0 (ピックリスト追加 / 必須フィールド追加 / 権限セット追加 等) |
| AI コスト | USD/PR | 0.50 USD 以下 (典型的な 50 ファイル PR) |

### 検証プロセス

1. 検証者が利用者プロジェクト (Dev Edition の yohaku-trial 拡張または別プロジェクト) で 2 つの差分を作る
   - データモデル変更 (新フィールド追加)
   - 自動化変更 (Flow 修正 or Trigger 追加)
2. `git diff` を取り、`/classify-diff` を起動
3. 分類結果を人手で評価:
   - データモデル分類が正しいか
   - 自動化分類が正しいか
   - 業務影響の推測が妥当か
   - 手動作業 (ピックリスト追加など) を見逃していないか
4. アンケート + retrospective 記録

## 再現性ガバナンスの本格稼働

本 Phase で初めて以下の機構が動く:

### 層 1: プロンプト = コード

- すべての分類 subagent system prompt を `prompts/` に配置 (Phase 3 中盤で `scaffold/.claude/agents/` から `prompts/` へ移動するか検討)
- SHA-256 ハッシュを change_summary に埋め込み

### 層 2: source 列必須化

- change_summary の各フィールドが `{ value, source, prompt_hash, model, temperature }` の wrapper 構造
- ajv で `source` 未指定をブロック

### 層 3: ゴールデンテスト

- `tests/golden/ai/classify-diff/` に入力 PR diff + 期待 change_summary 構造のペアを 3 件配置
- JSON Schema 適合 + キーフレーズ含有のチェック

### 層 4: 一致率 CI

- temperature=0 / 同一プロンプトハッシュ / 同一入力 で N=5 回再実行
- 完全一致率 80% 以上を CI でブロック条件
- 数値は Phase 7 で 95% に引き上げ

### 層 5: 決定的 vs AI の境界

- `scope_size` (行数 / ファイル数から計算) → deterministic
- `business_impact_hint` → ai
- `customer_communication_needed` → human (HUMAN_MANAGED)

## 公式エコシステム連携 (Phase 3 で実装)

- **Code Analyzer v5 SARIF 入力**: `yohaku diff --include-static-analysis report.sarif` で読み込み、`change_summary.static_analysis_findings` に格納 (詳細 ADR: [Code Analyzer SARIF 統合](./2026-05-07-code-analyzer-sarif-integration.md))
- **ApexGuru** (DX MCP scale-products toolset): logic-classifier から呼び出し、性能観点を委譲 (Phase 3 後半 or Phase 4)

## トレードオフ

- **代償**:
  - 8 サイクルは Phase 2 (8) と同等で、ボリューム大きい
  - 一致率 CI が初期は不安定で偽陽性が出るリスク → 80% を初期値、95% を Phase 7 ゴールに段階化
- **将来課題**:
  - 大規模 org / 大型 PR でのコスト最適化は Phase 7 (Haiku への退避ロジック) で詰める
  - subagent 並列起動の rate limit 対応は実測してから決める

## 代替案 (検討済み)

| 案 | 採否 | 理由 |
|---|---|---|
| A. **8 サイクル分割 (本案)** | 採用 | Phase 2 と同流儀、各サイクル独立検証可能 |
| B. SARIF 取り込みを Phase 4 に繰り延べ | 却下 | Phase 3 で並列分類器が動くタイミングと取り込みが噛み合うため |
| C. `/classify-diff` を 1 subagent でやる | 却下 | 並列性がなく、コンテキスト膨張のリスク。5 種に分割が分類精度面でも安全 |
| D. 一致率 CI を Phase 7 まで先送り | 却下 | Phase 3 で動かないと再現性ガバナンスが机上の空論になる |
