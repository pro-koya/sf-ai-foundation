# Phase 4 着手計画

## 判断

Phase 4 を以下 **7 サイクル** に分割して実行する。
Phase 3 の `change_summary` を直接の入力として、リリース事故の最大要因「手動作業の抜け」を構造的に解消する。

## 背景 (Phase 3 検証ゲートからの示唆)

Phase 3 検証ゲート (sfai-trial) で、以下の手動作業パターンが分類器によって既に発見されていた:

- **新規 PermissionSet 追加** → デプロイで assignment 自動付与されない (RiskOps)
- **新規 PermissionSet** → fieldPermissions が空 (要追記)
- **新規 Custom Field** → FLS / Layout / Validation Rule の追加検討
- **Risk_Tier__c (新規 Picklist)** → 既存レコードの値補正 (default 値が新規にしか効かない)

Phase 4 はこれらを **構造化された手動作業レジストリ** として持続化し、リリース時に「忘れない」仕組みを作る。

## サイクル構成

| # | サイクル名 | 主要成果物 | 完了条件 |
|---|---|---|---|
| 4-1 | manual_step スキーマ + 自動抽出ルール | `manual-step.schema.json`, `Tracked<T>` 拡張、ajv 強制 | スキーマで source 列必須が強制される |
| 4-2 | manual-step-extractor subagent | scaffold/.claude/agents/manual-step-extractor.md.eta | change_summary から手動作業候補を抽出 |
| 4-3 | release_doc スキーマ + 6 セクション仕様 | リリース対象 / 手動作業 / 事前事後確認 / Go-NoGo / ロールバック / 顧客周知 | スキーマ + テスト |
| 4-4 | release-composer + rollback-drafter subagent | 上記 6 セクションを組み立て、ロールバック手順は **ドラフトのみ** (人手承認前提) | scaffold 2 ファイル |
| 4-5 | `/release-prep` slash command + ops/registry/manual-steps-registry.md | --from --to で全 6 セクションを Markdown 化 | scaffold + 仕様 |
| 4-6 | `/manual-steps` slash command + 手動作業レジストリ | リリース横断で手動作業を集約 | scaffold |

各サイクル末に retrospective 必須。Phase 4 全体の期間目安: **2 週間相当**。

## 検証ゲート設計

### ゴール

利用者の sfai-trial で **1 つの mock リリース** を実施し:
- `/release-prep --from <previous-release-tag> --to HEAD` で 6 セクション Markdown 生成
- 手動作業見逃し件数 = 0
- ロールバック手順ドラフトが妥当 (人手で読んで Go と判断できる)

### 計測項目

| 指標 | 単位 | 目標 |
|---|---|---|
| `/release-prep` 完走時間 | 秒 | 90 秒以内 (5 並列 classifier + manual-step-extractor + release-composer の合計) |
| 手動作業見逃し件数 | 件 | 0 (Phase 3 検証ゲートで既に発見済の 4 パターン全カバー) |
| ロールバック手順の妥当性 | 5 段階 | 4 以上 (人手評価) |
| 顧客周知文の妥当性 | 5 段階 | 3 以上 (テンプレート埋め込み + 人手仕上げ前提なので) |
| AI コスト | USD/release | 1.00 USD 以下 |

### 検証プロセス

1. sfai-trial に previous-release tag (`git tag v0.0.0`) を baseline で打つ
2. Phase 3 検証ゲートで作った差分が `feat/risk-tier` として既存
3. `git tag v0.1.0` で現状を打つ
4. Claude Code から `/release-prep --from v0.0.0 --to v0.1.0`
5. 生成された `docs/releases/v0.1.0.md` を人手レビュー
6. retrospective に計測結果を記録

## 代替案

| 案 | 採否 | 理由 |
|---|---|---|
| A. **7 サイクル分割 (本案)** | 採用 | Phase 3 と同流儀。各サイクル独立検証可能 |
| B. release-prep を 1 つの巨大 subagent でやる | 却下 | コンテキスト膨張、再現性低下、責務分離原則に反する |
| C. ロールバック手順の完全自動化 | **却下** (重要) | 間違ったロールバック手順は事故を拡大する。Phase 4 では「ドラフト + 人手承認」を死守。完全自動化はしない |
| D. 顧客周知文の自由生成 | 却下 | ハルシネーションリスク。**テンプレート埋め込み方式**で自由生成を抑制し、最終的に HUMAN_MANAGED ブロックで人手仕上げ |

## トレードオフ

- **代償**:
  - 手動作業の **過剰検出 vs 見逃し** のバランス: 漏れは事故、過剰は「うるさい」。Phase 4 では **過剰検出を許容** (見逃しよりは安全)
  - リリースバージョニングの命名規則がプロジェクト依存: 設定可能にする (タグパターン `--release-tag-pattern`)
- **将来課題**:
  - 過去リリースとの整合性 (同じ変更が複数回リリースに乗ることの検出): Phase 7 課題
  - 自動デプロイ連携は **本 OSS の目的外** (sf CLI 領分、IMPLEMENTATION_GUIDE.md §非目的)
