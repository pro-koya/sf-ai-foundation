# Phase 8 着手計画

## 背景 (利用者からの指摘)

> 「各処理の詳細、ロジックの説明、フローチャートはまだない。今のままだとメタ情報に寄っていて、処理の中身がわからない。」

Phase 7 までで以下は揃った:

- 構造情報 (メソッド一覧 / SOQL 文字列 / DML / Trigger→Handler の Mermaid 概観)
- 自動懸念 (HIGH/MEDIUM/INFO)
- 上層部資料 (executive/summary, executive/risks)

不足しているのは「**個々の処理が何をしているのか**」を示すレイヤ。具体的には:

1. **Apex メソッドごとの制御フロー**: if/else / for / SOQL / DML / return が**どの順序で**走るかを図示
2. **Flow の decision 分岐**: 各要素から次要素への遷移を可視化
3. **自然言語のロジック説明**: 「このメソッドは何を計算しているか」「この Flow は何をしているか」

Phase 7 で AI_MANAGED マーカーは設置済みだが、**書き換えるエージェントは未実装**。

## 判断 — Phase 8 のスコープ

「**B (決定的) + A (LLM)** のハイブリッド」を採用する。理由は (1) 自然言語と制御フロー図は性質が異なる、(2) 制御フロー図は決定的に出せれば再現性が完璧、(3) 自然言語は人手 / LLM が補う領域、と分離が明確。

| 区分 | サブフェーズ | やること | 特性 |
|---|---|---|---|
| **B (決定的)** | 8-B1 | Apex メソッド本体の制御フロー抽出 (regex + brace counting) | 完全再現 |
| **B (決定的)** | 8-B2 | メソッド単位 Mermaid 生成 (1 メソッド 1 枚) | 完全再現 |
| **B (決定的)** | 8-B3 | Flow decision 分岐の Mermaid 展開 (XML から有向グラフ) | 完全再現 |
| **B (決定的)** | 8-B4 | yohaku-trial 中間検証 | 検証ゲート |
| **A (LLM)** | 8-A1 | `/yohaku-explain` slash command 設計 | Claude Code 連携 |
| **A (LLM)** | 8-A2 | AI_MANAGED ブロックを安全に更新する CLI ヘルパー | 決定的境界 |
| **A (LLM)** | 8-A3 | yohaku-trial 実機検証 (HUMAN_MANAGED 保護 / 再実行安定) | 検証ゲート |

## スコープに**入れない**もの (明示)

- LLM API 直叩き (OpenAI / Anthropic) — Phase 8 では Claude Code セッション内のみ
- 動的 SOQL 文字列の追跡 — AST が必要、Phase 9 候補
- メソッド間の inter-procedural 解析 — 今は intra のみ
- Apex AST ベースのリライト — regex 限界の前で停止する

## 設計原則 (Phase 8 で守る 5 つ)

1. **3 層分離は絶対**: 制御フロー Mermaid は DETERMINISTIC、ロジック説明は AI_MANAGED、業務文脈は HUMAN_MANAGED
2. **再現性ファースト (B)**: 同じ入力 → 同じ Mermaid。乱数も時刻もテンプレート以外には混ぜない
3. **AI 上書き境界の厳格化 (A)**: AI_MANAGED 以外には書かない。slash command の指示文に明文化
4. **段階検証**: B1→B2→B3→B4→A1→A2→A3 の各サイクルで yohaku-trial か unit test を必ず通す
5. **誤検出の取り扱い**: regex の限界はゼロにできない。誤検出は HUMAN_MANAGED で抑止する前提を明示
