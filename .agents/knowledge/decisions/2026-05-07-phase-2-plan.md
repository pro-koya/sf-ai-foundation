Phase 2 を以下の **8 サイクル** に分割して実行する。検証ゲート (現役プロジェクトでの `/onboard` 完走 + 定性 FB) を Phase 2 の **最終ゲート** として位置付け、サイクル末まで通過を確認しない。

## 検証ゲートの設計

### ゴール

Phase 2 の検証ゲートは **「Salesforce プロジェクトに参画したことがある 1 名が、本リポジトリの sfai を使って `/onboard` を完走し、所要時間と理解度を計測する」** ことを目的とする。

### 計測項目

| 指標 | 単位 | 目標 |
|---|---|---|
| `sfai init` から `/onboard` 完走までの所要時間 | 分 | 30 分以内 |
| 主要オブジェクト 5 件の理解度 (5 段階自己評価) | スコア | 平均 4 以上 |
| AI 任せられたタスク比率 | % | 60% 以上 |
| 完走中に発生したエラー / つまずき | 件 | 全件 `pitfalls/` へ記録 |
| 主観的ストレス指標 (NPS / 5 段階) | スコア | +1 以上 |

### 検証プロセス (1 セッション)

1. 検証者が利用者プロジェクトに `sfai init` を実行
2. `sf project retrieve start` で最新メタデータを取得済みであること
3. `sfai graph build` 実行
4. Claude Code で `/onboard` 起動
5. 主要オブジェクト 5 件について `/impact` `/explain` を試す
6. セッション終了後、5 分以内にアンケート記入
7. 結果を `.agents/knowledge/retrospectives/` に記録

### 検証プロジェクトの選定

ユーザが現在参画中の Salesforce プロジェクトを使用する想定。代替として:

- (a) 利用可能な `examples/sample-project/` (Phase 2 サイクル 2-7 で構築)
- (b) Salesforce Trailhead の DreamHouse / E-Bikes のいずれか (公開ダミーデータ)

最初の検証は (a) または (b) で動作確認し、その後利用者の実プロジェクトで本番検証する 2 段構えを推奨する。

## 代替案 (検討済み)

| 案 | 採否 | 理由 |
|---|---|---|
| A. **8 サイクル分割 (本案)** | 採用 | 各サイクルの完了条件が独立検証可能、検証ゲートを最後に置ける |
| B. ビッグバン実装 (1 ターンで全部) | 却下 | Phase 1 で経験済みだが、実環境動作未確認のリスクが残った。Phase 2 ではより慎重に |
| C. 検証ゲートを各サイクルに分散 | 却下 | 検証コスト (人間の時間) が高すぎる、Phase 5 で persona 別検証する方が経済的 |
