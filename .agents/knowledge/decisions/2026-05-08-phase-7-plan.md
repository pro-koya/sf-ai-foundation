# Phase 7 着手計画

## 判断

Phase 7 を **5 サブフェーズ** に分割。利用者からの要望「資料とソースの乖離を OSS で構造的に解消したい」「上層部・承認者にも届く資料が欲しい」を本 Phase の **核心目的** として明確化する。

## 北極星

> **force-app/ にあるあらゆる Salesforce 構成要素を、誰が見ても (開発者・管理者・上層部・監査担当・顧客) 役に立つ Markdown として常に最新の状態で提供する。**

これは IMPLEMENTATION_GUIDE.md の中核目的「属人化解消」「運用タスク自動化」を **ドキュメント問題** に焦点を当てて完遂する。

## サブフェーズ構成

### Phase 7-A: 既存知識グラフ全種を Markdown 化 (4 サイクル)

| # | 対象 | 出力 |
|---|---|---|
| 7-A1 | Flow | `docs/generated/flows/<Name>.md` |
| 7-A2 | ApexClass | `docs/generated/apex/<Name>.md` |
| 7-A3 | ApexTrigger | `docs/generated/triggers/<Name>.md` |
| 7-A4 | PermissionSet / Profile / ValidationRule (個別) | `docs/generated/{permissions,validation-rules}/<Name>.md` |

各 Markdown は 3 種ブロック (DETERMINISTIC / AI_MANAGED / HUMAN_MANAGED) 構造で、`sfai render --all` または各個別 target で再生成。

### Phase 7-B: 取り込み拡充 (Salesforce 構築上重要なメタデータ全網羅) (10 サイクル目安)

| # | 対象 | 重要度 | 業務価値 |
|---|---|---|---|
| 7-B1 | **ApprovalProcess** (承認プロセス) | ★★★ | 業務フローの中核、レビューに必須 |
| 7-B2 | **RecordType + GlobalValueSet** (レコードタイプ / グローバル値セット) | ★★★ | データモデルの分岐、業務ロジック分岐の起点 |
| 7-B3 | **Layout / FlexiPage** (画面構成) | ★★★ | UI 配置、Field-Level Security の効果検証に必須 |
| 7-B4 | **CustomMetadataType / CustomLabel** (設定の中核) | ★★★ | 環境別設定、多言語対応 |
| 7-B5 | **DuplicateRule / MatchingRule** (データ品質) | ★★ | 重複防止 |
| 7-B6 | **AssignmentRule / EscalationRule / AutoResponseRule** (Case 系自動化) | ★★ | サポート業務 |
| 7-B7 | **Workflow / EmailTemplate** (レガシー自動化 + 通知) | ★★ | レガシー → Flow 移行検出に必須 |
| 7-B8 | **NamedCredential / RemoteSiteSetting / ConnectedApp** (統合点) | ★★★ | セキュリティ監査の対象 |
| 7-B9 | **PlatformEvent / Custom Notification / Outbound Message** (イベント駆動) | ★★ | 統合アーキテクチャ |
| 7-B10 | **LWC / Aura 構造解析** (子コンポーネント参照、Apex 呼び出し) | ★★★ | Phase 1 既知制約の解消 |

各サイクルで:
- 知識グラフ拡張 (extractor + SQLite テーブル)
- Markdown render テンプレ
- Phase 7-D で改善案ヒントの自動生成対象に組み込み

### Phase 7-C: 横断ドキュメント (人間の理解を加速) (4 サイクル)

| # | 横断ドキュメント | 出力 | 価値 |
|---|---|---|---|
| 7-C1 | ER 図 (Mermaid) | `docs/generated/diagrams/erd.md` | データモデル一覧 |
| 7-C2 | 自動化マトリクス | `docs/generated/automation-matrix.md` | オブジェクト × Trigger/Flow/Workflow の発火表 |
| 7-C3 | 権限マトリクス | `docs/generated/permission-matrix.md` | PermissionSet/Profile × Object/Field × CRUD/FLS |
| 7-C4 | 依存グラフ + ヘルスレポート | `docs/generated/diagrams/dependency-graph.md`, `health-report.md` | Apex/Flow/Trigger 参照関係、Naming 違反、未使用検出 |

すべて Mermaid + Markdown で出力。GitHub / Salesforce DX エコシステムで そのままレンダリング可能。

### Phase 7-D: AI による「気になる点・改善案」自動生成 (3 サイクル)

各エンティティ Markdown に **`<!-- AI_MANAGED_START id="concerns" -->`** ブロックを追加し、AI が懸念点を提示:

- **Flow**: 再帰実行リスク、無限ループ、未処理エラーパス
- **ApexClass**: テストカバレッジ不足、Bulk 安全性、shared mutable state
- **ApexTrigger**: 1 オブジェクト複数トリガー、エンタープライズパターン違反
- **PermissionSet**: 未割当、過剰権限の可能性
- **CustomField**: 命名規則違反、reportable=false で集計不可
- **Layout**: 必須フィールドが未配置
- **承認プロセス**: 承認者未指定、循環参照リスク

Code Analyzer SARIF (Phase 3 連携済) と知識グラフを組み合わせて自動表示。

### Phase 7-E: 上層部・承認者向け資料 (4 サイクル) ★★★ 利用者要望の中核

| # | 資料 | 想定読者 | 出力 |
|---|---|---|---|
| 7-E1 | **Executive Summary** (1 ページサマリ) | CEO / CIO / CTO / 事業責任者 | `docs/generated/executive/executive-summary.md` |
| 7-E2 | **Architecture Overview** (アーキテクチャ概要) | 上位アーキテクト / IT 部門責任者 | `docs/generated/executive/architecture-overview.md` |
| 7-E3 | **Security & Compliance Posture** (セキュリティ・コンプライアンス姿勢) | セキュリティ責任者 / 監査担当 / 法務 | `docs/generated/executive/security-posture.md` |
| 7-E4 | **Risk & Change Impact Board** (リスク・変更影響ボード) | 承認者 / プロジェクトオーナー / リリース承認会議 | `docs/generated/executive/change-impact-board.md` |

#### 7-E1: Executive Summary の構造例

```markdown
# <%= projectName %> Salesforce 実装サマリ

> 自動生成 (sfai render executive)。<lastUpdated> 時点。

<!-- DETERMINISTIC_START id="scale" -->
## 実装規模
- ビジネスドメイン: 営業 / 経理 / サポート (3 ドメイン)
- データモデル: <objectsCount> オブジェクト、<fieldsCount> フィールド
- 自動化: <triggersCount> トリガー / <flowsCount> Flow / <approvalsCount> 承認プロセス
- カスタムロジック: <apexClasses> クラス (<linesOfCode> 行)
- 権限管理: <permissionSets> PermissionSet / <profiles> Profile
- 統合点: <integrations> 件 (NamedCredential / RemoteSite / Connected App)
<!-- DETERMINISTIC_END id="scale" -->

<!-- AI_MANAGED_START id="business-value" -->
## ビジネス価値の概要 (AI 推測、業務用語)
- 何を実現しているか (3 行で)
- 誰が使っているか (営業 N 名、経理 M 名 等)
- どの業務プロセスを自動化しているか
<!-- AI_MANAGED_END id="business-value" -->

<!-- AI_MANAGED_START id="recent-changes" -->
## 直近の変化 (直近 90 日)
- 主要な変更点 (技術用語抜き)
- 業務影響
- 今後の予定
<!-- AI_MANAGED_END id="recent-changes" -->

<!-- HUMAN_MANAGED_START id="strategic-context" -->
<!-- ここに事業戦略との接続を人手記述 (例: 中計とのアラインメント) -->
<!-- HUMAN_MANAGED_END id="strategic-context" -->
```

#### 7-E2: Architecture Overview

- ER 図 + 説明 (一般用語で)
- レイヤー構成図 (UI / 自動化 / 統合 / データ / 権限)
- 主要な統合点
- スケーラビリティ・性能観点 (Apex 行数、Flow ノード数等の規模感)

#### 7-E3: Security & Compliance Posture

- 権限マトリクス概要 (誰が何にアクセスできるか)
- データ機密度分類 (`secrets-rules.yaml` ベース)
- 共有モデル (Sharing Rule / Owner-based Sharing)
- 認証情報の管理状況 (NamedCredential 一覧、ハードコード Secret 検出)
- PII 含有フィールド一覧 (監査用)

#### 7-E4: Risk & Change Impact Board

- 直近変更サマリ (`docs/ai-augmented/change-summaries/` から集約)
- 既知の技術的負債 (Code Analyzer findings 集計、Workflow → Flow 移行候補等)
- 単一障害点候補 (依存グラフの中心ノード)
- 廃止予定機能の利用状況 (Workflow / Process Builder 使用検出)
- リリース承認の判断材料 (Phase 4 の release_doc を経営層向けに翻訳)

### Phase 7-F: 普及 (継続) — 別途運用フェーズ

OSS 公開・事例蓄積・コミュニティ運営は本計画 ADR では **継続的活動** として扱い、サイクル化しない。Phase 7-A〜E が完了した時点で OSS 公開判断。

## 検証ゲート設計

### サブフェーズごとに検証ゲートを設定

- **7-A 末**: sfai-trial で全種 Markdown 生成、利用者が `docs/generated/` を見て「網羅されている」と確認
- **7-B 末**: 拡充メタデータが知識グラフ + Markdown に反映、サンプルプロジェクトで動作
- **7-C 末**: ER 図 + 自動化マトリクス + 権限マトリクスが GitHub プレビューで読める
- **7-D 末**: 「気になる点」AI 出力の精度評価 (人手照合)
- **7-E 末**: **利用者上長 / 承認者役の人に資料を見せて理解度を計測** (5 段階自己評価)

### Phase 7 全体 (7-F 進入条件)

- 全 Markdown が `sfai sync` 1 コマンドで再生成できる
- HUMAN_MANAGED ブロックが全て保護される
- 大規模 org (5000+ オブジェクト) で性能目標を満たす (Phase 7 後半で測定)
- 利用者が「資料の更新がいらない、ソースが正本」と実感できる

## 主要設計判断

### `sfai render` の対象拡張

```bash
# 全種一括
sfai render

# 個別
sfai render flows
sfai render apex
sfai render triggers
sfai render permissions
sfai render validation-rules

# Phase 7-B 拡充後
sfai render approvals
sfai render layouts
sfai render record-types
sfai render custom-metadata
...

# 横断ドキュメント (Phase 7-C)
sfai render erd
sfai render automation-matrix
sfai render permission-matrix
sfai render dependency-graph
sfai render health-report

# 上層部資料 (Phase 7-E)
sfai render executive       # 4 ファイルまとめて
sfai render executive-summary
sfai render architecture-overview
sfai render security-posture
sfai render change-impact-board
```

### `--with-concerns` フラグ (Phase 7-D)

各エンティティ Markdown に AI 「気になる点」セクションを含めるかどうかを制御:

```bash
sfai render flows --with-concerns
```

AI 呼び出しを伴うため `sfai metrics record` で記録、コスト管理。

### 上層部資料の業務翻訳

`customer-impact-explainer` (Phase 5) の延長として、**executive-summarizer** subagent を新設:

- 技術用語 → 経営用語
- 規模感を「N オブジェクト」ではなく「営業全プロセスをカバー」のように
- 数値を文章に (規模の大きさを伝える)

## 代替案

| 案 | 採否 | 理由 |
|---|---|---|
| A. **5 サブフェーズ分割 (本案)** | 採用 | 各サブフェーズ独立検証可能、利用者要望「全メタデータ + 上層部資料」を完全カバー |
| B. 7-E (上層部資料) を Phase 7 完了後に別途 | 却下 | 利用者の核心要望、Phase 7 内で実装すべき |
| C. 7-B のメタデータ拡充を全部やる | 部分採用 | 重要度 ★★★ から着手、★★ は時間に応じて |
| D. Mermaid 図を SVG 生成まで | 却下 | Markdown 内 Mermaid で GitHub プレビュー可、SVG は Phase 8 で検討 |
