# Phase スコープ規律の確立とリリース計画の整流化

## 背景 (なぜこの判断が必要か)

2026-05-08 に Phase 7 を「普及」から「ドキュメント完全化 + 上層部資料 (5 サブフェーズ 25 サイクル)」に再定義した時点から、**新 Phase の歯止めが効かない状態**が続いていた。

2026-05-10 にオーナーから棚卸し依頼があり、実態を確認したところ以下が明らかになった。

- IMPLEMENTATION_GUIDE.md v1.1 の元計画は **Phase 1〜7 で完結** する想定だった
- しかし 2026-05-08 〜 05-09 のわずか 2 日間に Phase 7 → 8 → 9 → 9.x → 10 → 11 → 12 → 13 → 14 → 15 と **9 個の新 Phase が連鎖追加** された
- これら Phase 8〜15 は本質的にすべて「Phase 7 (ドキュメント完全化)」の中で発見された課題の繰越であり、本来 Phase 7 内で責任を持って対処すべきものだった
- Phase 15 に至っては **「コード変更ゼロ・検証のみ」** で 1 つの Phase を立てており、本来 Phase 14 の検証ゲートに含まれるべきものだった
- CHANGELOG が v0.1.0 (Phase 1〜6) で停止していた事実も、**「Phase 7 がいつまでも閉じない」ことの帰結** だった
- 本来の北極星「内部検証 (現参画プロジェクトでの実利用実証)」は **未達のまま、技術拡張だけが進行していた**

オーナーからの指摘:

- 「目的とゴールが曖昧になっている」
- 「Phase 内で見つかった問題は新 Phase に逃さず、その Phase の責任で対処してほしい」
- 「むやみに Phase を増やさないでほしい (計画の乱れに繋がる)」
- 「本当に価値あるものを目指して着実に進もう」

## 判断 (決定したこと)

### 判断 1: Phase スコープ規律の明文化

`AGENTS.md` に新章「Phase スコープ規律」を新設し、以下を強制する。

1. **スコープロック**: Phase 着手時に ADR で「北極星」「ゴール条件 (DoD)」「Out of Scope」「対応リリース」を明記し、以後追加しない
2. **デフォルトは Phase 内対処**: 進行中に見つかった課題は原則そのPhase内で責任を持って解決する。Phase が長くなることを許容する
3. **新 Phase 追加の閾値**: 以下の (a) (b) を **両方** 満たす場合のみ起案を許可する
   - (a) 現 Phase の北極星目標から **明確に外れている**
   - (b) **人間オーナーの明示承認** がある
4. **新 Phase 着手 ADR の必須項目**: 「なぜ前 Phase の中ではなく独立 Phase である必要があるか」を必ず記述する
5. **検証だけで Phase を立てない**: 検証ゲートは元の Phase の完了条件に含める

### 判断 2: リリース計画の整流化 — Phase 7〜15 を v0.2.0 として統合

過去の Phase 7〜15 は、本来 1 つの Phase として完結すべきだった「ドキュメント完全化 + AI 文面生成基盤」というテーマだった。これを **v0.2.0 リリースの構成要素** として統合扱いに整理する。

新しいリリース計画 (3 段階ゴールに対応):

| リリース | 段階 | テーマ | 状態 |
|---|---|---|---|
| v0.1.0 | 内部検証 (基盤) | 知識グラフ + CLI + Claude統合 + 差分意味づけ + 手動作業管理 + オンボーディング + Plugin化 | ✅ リリース済 (Phase 1〜6) |
| v0.2.0 | 内部検証 (拡充) | ドキュメント完全化 (21 メタデータ種対応) + 処理フロー可視化 + AI 文面生成基盤 (`/yohaku-explain`) | ✅ 本判断と同時にリリース宣言 (旧 Phase 7〜15 を統合) |
| **v0.3.0** | **内部検証 (実証)** | **現参画プロジェクトでの実利用 + KPI 計測 + 既知 pitfalls の解消** | **次に着手** |
| v0.4.0+ | 社内展開 | 社内他プロジェクトへの展開、フィードバック反映 | 将来 |
| v1.0.0+ | 社外展開 | OSS 公開、コンサル導入、コミュニティ運営 | 将来 |

### 判断 3: 「内部検証完了」の Definition of Done を明示

これまで「内部検証」の DoD が曖昧だったため技術拡張が止まらなかった。以下を明示的な完了条件とする (IMPLEMENTATION_GUIDE.md の人間側 KPI 5 項目をベースに、現実的な閾値で再設定)。

| 完了条件 | 閾値 | 計測方法 |
|---|---|---|
| 現参画プロジェクトで `yohaku sync` を週次運用 | 4 週連続 | 運用ログ |
| 利用者主観: 「資料更新が不要、ソースが正本」と実感 | 5 段階で 4 以上 | 振り返りヒアリング |
| 運用タスク AI 任せ可能比率 | 60% 以上 | 1 サイクルの作業ログ分類 |
| 1 機能あたりリリース準備時間 | 50% 短縮 | リリース前作業の前後比較 |
| 手動作業見逃し件数 | 0 / リリース | リリース後インシデント数 |

**5 項目中 3 項目以上で達成** が「内部検証完了 = v1.0.0 候補入り」の条件。これを満たすまでは技術拡張ではなく実利用に注力する。

### 判断 4: 既存ナレッジは履歴保全 (改竄しない)

`.agents/knowledge/decisions/` の Phase 1〜15 ファイルは **改竄せず履歴として残す**。これらは「実際に何が起きたか」の正本記録であり、今回の規律確立の出発点でもある。リネームも行わない。

代わりに本 ADR と IMPLEMENTATION_GUIDE.md / CHANGELOG.md の更新で **「これからのリリース計画」** を明確化する。INDEX.md には本 ADR を追加し、過去の Phase ADR との関係を明示する。

### 判断 5: 禁則 14 の追加

`IMPLEMENTATION_GUIDE.md` の禁則一覧に以下を追加する。

> **禁則 14: Phase の安易な増殖禁止**
> - Phase 内で発見された課題を新 Phase に逃がさない (原則 Phase 内で対処)
> - 新 Phase 追加には判断 1 の閾値 (現 Phase 目標から明確に外れる + 人間承認) を満たすこと
> - 検証だけで Phase を立てない (検証は元の Phase の完了条件に含める)

## 代替案

| 案 | 採否 | 理由 |
|---|---|---|
| A. 規律明文化 + リリース計画整流化 (本案) | 採用 | オーナー要望に沿う。過去履歴を保全しつつ将来の暴走を抑止できる |
| B. Phase 7〜15 のナレッジファイルもリネーム/統合 | 却下 | 履歴改竄リスク、リンク切れ、整合性確認コスト大 |
| C. 規律だけ追加、リリース計画は据え置き | 却下 | CHANGELOG と Phase の乖離が解消されない |
| D. 元の Phase 1〜7 構造を完全廃止し新体系に作り変え | 却下 | 過剰な変更、既存 ADR との整合維持コスト大 |

## トレードオフ

- **代償**:
  - リリース計画と過去 Phase の二重命名期間が発生する (ナレッジは Phase 1〜15、リリースは v0.1.0 / v0.2.0)
  - これからは「Phase」という言葉を慎重に使う必要がある (実装サイクル名 vs リリースマイルストーン)
  - v0.3.0 (内部検証) に振り切ることで、既知 pitfalls (SOQL 検出漏れ等) の修正タイミングが遅れる可能性 → ただし pitfalls は v0.3.0 内で対処するため問題ない (これが新規律の最初の実例になる)
- **将来課題**:
  - v0.3.0 完了時点で本 ADR の規律が機能したか振り返り (`retrospectives/2026-XX-XX-v0.3.0-discipline-review.md`)
  - 規律が機能しなければ、より強い仕組み (ADR テンプレート強制、CI による Phase 着手前チェック) を検討
