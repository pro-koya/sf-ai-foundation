# Phase 9 着手計画

## 背景 (利用者からの指摘)

> 「フローチャートの各要素の文字列が多くなると途切れてしまって、中身がわからなくなる」
> 「処理概要については、自然言語でわかりやすく、かつ処理がざっくりわかるように、書いてほしい。パッとわかるようにしてほしい」
> 「Apex/Flow に限らず、承認プロセスや共有ルール、入力規則など条件が絡むメタデータも対象にしてほしい」

Phase 8 で Mermaid 化と `/sfai-explain` の経路は確立したが:

1. ノードラベルを 60 文字で truncate しているため、長い SOQL や式が `…` で切れる
2. AI_MANAGED が空のままだとパッと意図が掴めない (`/sfai-explain` を毎度走らせないと)
3. ValidationRule の formula は raw 文字列でしか表示されておらず、条件の意味が伝わらない
4. ApprovalProcess / SharingRules は取り込みすら未実装

## 判断 — Phase 9 のスコープ

「**今回サイクルでは品質を担保できる範囲に絞る**」方針を採用する。理由は (1) 利用者の念入り検証ご要望、(2) sfai-trial に ApprovalProcess / SharingRules の fixture が無く検証品質が下がる、(3) 9-A 系は全エンティティに横串で効くので投資対効果が大きい。

| 区分 | サブ | やること |
|---|---|---|
| **A 横串改善 (今回)** | 9-A1 | フローチャートに **ノード詳細表 (Step ID → 全文)** を併記。Mermaid は短く、表で完全な情報を提示 |
| | 9-A2 | 全エンティティ最上部に **Quick Summary (DETERMINISTIC, 1〜3 行)** を追加 |
| **B 条件式 (今回)** | 9-B1 | ValidationRule の `errorConditionFormula` を簡易 parser で AND/OR/NOT/leaf に分解し、自然語に変換 |
| **C 検証 (今回)** | 9-C | sfai-trial で実機検証、完了 ADR、INDEX 更新 |

## スコープ外 (後続)

- **9-B2: ApprovalProcess 取り込み** — sfai-trial に fixture 無し。Phase 7-B 拡張パターンで後続着手
- **9-B3: SharingRules 取り込み** — 同上
- **9-B4: PermissionSet の権限可視化** — フィールド/オブジェクト権限のマトリクス化、後続
- **AI_MANAGED への自動説明生成** — `/sfai-explain` を CI 化する話。Phase 10 候補

## 設計

### 9-A1: ノード詳細表

`buildMethodFlowchart` の戻り値を拡張:

```ts
interface MermaidFlowchart {
  readonly mermaid: string;
  readonly details: readonly NodeDetail[];
}

interface NodeDetail {
  readonly id: string;        // "n5"
  readonly kind: string;       // "soql" / "for" / ...
  readonly label: string;      // Mermaid 内に出した短縮ラベル
  readonly fullText: string;   // 切られていない元テキスト
}
```

テンプレートは Mermaid の直後に Markdown 表 (Step / 種別 / 内容) を出す:

```markdown
```mermaid
...
```

| Step | 種別 | 内容 |
|---|---|---|
| n1 | if | `accountIds == null \|\| accountIds.isEmpty()` |
| n7 | for | `AggregateResult ar : [SELECT Account__c acc, SUM(Total_Amount__c) total FROM Invoice__c WHERE Account__c IN :accountIds AND IsPaid__c = false GROUP BY Account__c]` |
```

Mermaid 内のラベルは引き続き短く (truncate を 80 文字に微増)、不足分は表で補う。

`buildFlowFlowchart` も同じ拡張をする。

### 9-A2: Quick Summary

新モジュール `src/render/summary.ts`:

```ts
export function summaryForApex(cls: ApexClass, graph: KnowledgeGraph): string;
export function summaryForApexTrigger(trg: ApexTrigger, graph: KnowledgeGraph): string;
export function summaryForFlow(flow: Flow): string;
export function summaryForValidationRule(vr: ValidationRule): string;
export function summaryForRecordType(rt: RecordType): string;
export function summaryForPermissionSet(ps: PermissionSet): string;
```

各テンプレに `<!-- DETERMINISTIC_START id="quick-summary" -->` を追加し、決定的な短文を入れる。

例:

- Apex: 「**5 メソッド / SOQL 2 件 / DML 1 件** を含む `public with sharing` クラス。テストではない。コールアウト無し。」
- Flow: 「**`Order__c` の AfterSave 起点** で `Shipment__c` を 1 件作成する `AutoLaunchedFlow`。要素 1 / 分岐 0。」
- ValidationRule: 「**Active な入力規則**。条件式は AND 2 個、対象オブジェクト `Order__c`。」

### 9-B1: ValidationRule formula 自然語化

簡易 formula parser:

- 関数呼び出し: `AND(...)`, `OR(...)`, `NOT(...)`, `ISBLANK(...)`, `ISCHANGED(...)`, `ISNEW()`, `ISPICKVAL(...)`, `LEN(...)`, etc.
- フィールド参照: `Field__c`, `Object.Field`
- リテラル: `"x"`, `'x'`, 数値, `TRUE`, `FALSE`
- 比較: `=`, `!=`, `<>`, `<`, `>`, `<=`, `>=`

ツリーに分解し、leaf を自然語に変換:

| Salesforce 式 | 自然語 |
|---|---|
| `AND(a, b)` | 「以下を**全て**満たす: …」 |
| `OR(a, b)` | 「以下のいずれかを満たす: …」 |
| `NOT(a)` | 「以下を満たさない: …」 |
| `ISCHANGED(F)` | 「`F` が変更された」 |
| `ISBLANK(F)` | 「`F` が空である」 |
| `ISNEW()` | 「新規レコード作成時」 |
| `ISPICKVAL(F, 'V')` | 「`F` が 'V' である」 |
| `F = 'V'` | 「`F` が 'V' に一致する」 |
| `F < N` | 「`F` が N 未満」 |

未対応の関数は raw 表記で留める。誤検出は HUMAN_MANAGED で抑止できる前提。
