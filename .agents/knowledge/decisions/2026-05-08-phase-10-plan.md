# Phase 10 着手計画

## 背景

Phase 9.x で「条件絡みのメタデータ」(ValidationRule / ApprovalProcess / SharingRules) と「権限」(PermissionSet 詳細マトリクス) が完成。残スコープから「実用頻度が高く、構築物の多くに必ず登場するもの」を 4 種選定し、Phase 10 として進める。

## 判断 — Phase 10 のスコープ

| 区分 | サブ | やること | 優先理由 |
|---|---|---|---|
| **権限** | 10-A | Profile body 拡充 (PermissionSet と同等の権限マトリクス) | 多くの組織で Profile が今も主要な権限境界。現状はメタ情報のみ |
| **UI** | 10-B | Layout 取り込み | UI レイアウトはあらゆる案件に必ず存在。引き継ぎ時の情報量が大きい |
| **設定外出し** | 10-C | CustomMetadataType + Records 取り込み | 設定駆動開発のベストプラクティス。可視化されないと「どこに何が定義されているか」が消失 |
| **連携設定** | 10-D | NamedCredential + RemoteSiteSetting 取り込み | 外部連携の存在を承認者・運用者が把握できる必要がある (シークレットは出さない) |

## スコープ外 (Phase 11 以降)

- **LWC / Aura** — 多ファイル構造 (HTML + JS + CSS + .js-meta.xml) で取り込み構造が他と異なる。専用フェーズで設計
- **FlexiPage / Lightning App** — Layout の延長線。Phase 10 後に着手
- **再現性 CI** — `/yohaku-explain` の温度 0 / プロンプトハッシュ / N-run 一致テスト
- **Profile の field-level security ベスト pickList ベース照合** — 複雑なため別フェーズ

## 設計原則 (継続)

1. **3 層分離** (DETERMINISTIC / AI_MANAGED / HUMAN_MANAGED) を踏襲
2. **Quick Summary** を全エンティティに必ず付ける
3. **Mermaid は短いラベル + ノード詳細表** (Phase 9-A1 で確立した形)
4. **シークレット保護**: NamedCredential / RemoteSiteSetting で実値 (パスワード/トークン/エンドポイントのうち機密性の高い部分) は **存在の有無だけ** を出力。XML に書かれていても出さない
5. **fixture は実装前に yohaku-trial に追加** してから検証する (品質ゲート)

## ゴール条件

| # | 条件 |
|---|---|
| 1 | Profile が PermissionSet と同等の権限マトリクスで描画される |
| 2 | yohaku-trial に Layout / CustomMetadata / NamedCredential / RemoteSiteSetting の各 fixture を追加し、`yohaku sync` で正しく描画 |
| 3 | NamedCredential 出力でシークレットが文字列として漏れていないこと |
| 4 | 全テスト pass (新規 +N 件、目安 +20 件以上) |
| 5 | Phase 10 完了 ADR + INDEX 更新 |

## 段階的進行

```
10-A (Profile) → unit + yohaku-trial fixture (Profile 1 件追加) → 検証
10-B (Layout)  → unit + yohaku-trial fixture (Layout 1 件追加) → 検証
10-C (CMT)    → unit + yohaku-trial fixture (CMT type + 2 records) → 検証
10-D (Named)  → unit + yohaku-trial fixture (NC 1 件 + RSS 1 件) → 検証
完了 ADR + INDEX
```

各サブで yohaku-trial の `yohaku sync` を必ず通す。
