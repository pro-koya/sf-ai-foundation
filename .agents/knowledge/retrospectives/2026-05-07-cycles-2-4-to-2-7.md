# Retrospective: サイクル 2-4 〜 2-7

## サイクル要約

連続実装で 4 サイクル分を完了:

- **2-4**: hooks 仕様を実機計測。Node CLI cold-start で 600ms 消費するため 500ms 目標は非現実的と判明 → wall-clock 2 秒目標に調整
- **2-5**: `sfai metrics` 実装 (`record` / `show`、`.sfai/metrics.json` スキーマ、コスト推定、period 別集計)
- **2-6**: `sfai init` 実装 (scaffold/ → 利用者プロジェクトに eta 展開 + profile filter)
- **2-7**: `examples/sample-project/` (Customer__c / Order__c / OrderTrigger / CustomerService の 6 メタデータ) + `docs/01-getting-started/{quickstart, dev-edition-setup, profiles}.md`

CI 緑のまま (54 テスト pass)。E2E スモークで全コマンド動作確認済み。

## 1. 要件整理 — 学び

- 4 サイクルを 1 ターンで連続実行する選択は、IMPLEMENTATION_GUIDE.md の Phase 2 計画 (8 サイクル) のうち実装サイクルを束ねたもので、効率的だった
- ユーザの「Dev Edition で検証する」要望を受けて、検証ゲート (2-8) の前段として **詳細な手順書** を作るタスクが自然に追加された

## 2. 計画立案 — 学び

- 実機計測を最初に行う設計 (2-4 で hooks 計測) は良かった: 当初 500ms 目標が非現実的と早期に判明
- `sfai metrics` の `record` と `show` を分離した設計は再利用性が高い (Phase 5 の subagent 経由記録に流用可能)
- `sfai init` を eta + profile filter のみに絞ったことで Phase 2 スコープを守れた (interactive wizard は Phase 6 へ繰越)

## 3. 実行 — 学び

- `MAX_EVENTS_RETAINED = 5000` で metrics.json が無限肥大しないようガードした
- scaffold/ を packages/sfai-core/dist/scaffold/ にバンドルする copy-assets.mjs 拡張により、`sfai init` がインストール先のみで動く設計になった
- `sample-project` のテストデータは 6 ファイル (Customer + Order + Lookup + Currency + Picklist + Trigger) で依存解析の最低検証セットを満たす

## 4. レビュー — 学び

- **CRITICAL ゼロ** ✓
- **HIGH 警戒点**:
  - `sfai metrics record` は subagent から呼ばれる想定だが、認証や rate limit のチェックは無し → Phase 5 で hooks 経由 + sandbox 制限を追加検討
  - `sfai init --conflict skip` がデフォルトのため、再 init で新コマンドが追加されない罠あり → docs に明記済み
- **MEDIUM**:
  - `examples/sample-project/` のメタデータが最低限、Phase 6 でもう少し充実させる余地あり

## 5. 修正・再実装 — 学び

- 大きな手戻りなし
- ただしビルドアセットコピー (copy-assets.mjs) を 2 度拡張した (schema → templates → scaffold) ので、最初から動的に scan するロジックにすれば良かった (Phase 7 リファクタ候補)

## 6. 整理 — 学び

- `docs/01-getting-started/` に 3 つのドキュメントを配置:
  - `quickstart.md` — 30 分で動かすクイックスタート
  - `dev-edition-setup.md` — Dev Edition 取得から検証ゲートまでの詳細手順
  - `profiles.md` — minimal / standard / full の選び方
- README からの導線は次サイクル (2-8 完了後) で `docs/` に追加

## 良かった点 (Keep)

- **実機計測ファースト**: 推測で 500ms 目標を堅持せず、Node 実起動を測って 2 秒に修正したのは現実的
- **metrics の `record/show` 分離**: 単一責任、Phase 5 で subagent 経由記録に流用可能
- **scaffold バンドル戦略**: `dist/scaffold/` を同梱することで、`sfai init` が外部ファイル参照不要
- **sample-project の最小最大化**: 6 ファイルで依存解析の検証が完結 (Customer Lookup、Trigger、Picklist、Currency)
- **dev-edition-setup.md の網羅性**: Dev Edition 取得から検証ゲートまでステップ別、トラブルシュート、フィードバック記録形式を全部書いた

## 課題 (Problem)

- `sfai metrics record` を subagent から呼ぶ標準フックがまだ無い → Phase 5 課題
- Dev Edition setup ガイドは未検証 (ユーザの実体験フィードバック待ち)
- `sfai init` の interactive モードが無い (CLI flag 必須) → Phase 6 候補
