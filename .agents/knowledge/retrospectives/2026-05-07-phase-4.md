# Retrospective: Phase 4

## サイクル要約

手動作業管理 + リリース準備の 7 サイクルを yohaku-trial 実プロジェクトで完走。検証ゲート通過。

- 4-1: manual_step / release_doc スキーマ + Tracked&lt;T&gt; 拡張 + ajv 強制
- 4-2: ルールベース extractManualSteps + manual-step-extractor subagent
- 4-3: release_doc 6 セクション仕様 (JSON Schema)
- 4-4: release-composer + rollback-drafter subagent
- 4-5: /release-prep slash command
- 4-6: /manual-steps slash command
- **4-7: yohaku-trial の v0.0.0 → v0.1.0 で実機完走、Markdown 145 行生成**

CI 87/87 pass。実機で 4 件の手動作業を漏れなく検出。

## 1. 要件整理 — 学び

- Phase 3 検証ゲートで予測した 4 つの手動作業パターン (FLS / assignment / バックフィル / fieldPermissions) を Phase 4 で **構造化抽出として実装** できた
- ルールベース (core 内) と subagent ベース (補完用) の **2 段抽出** に分けた設計が機能的にきれいに収まった

## 2. 計画立案 — 学び

- 7 サイクル分割は Phase 3 と同流儀で安定
- **rollback-drafter を独立 subagent** にした判断は正解 — release-composer に混ぜると "完全自動化禁止" の境界が曖昧になる
- 「過剰検出 > 見逃し」の原則をルールベースに直接エンコードできた (extract-manual-steps.ts)

## 3. 実行 — 学び

- ルールベース 4 パターン + subagent (manual-step-extractor) で id 帯 (1〜99 vs 100〜199) を分離 → 重複しない
- release-composer + rollback-drafter を **並列起動** したことで合計 130 秒に収まった (順次なら 200 秒+)
- Tracked&lt;T&gt; / source 列を Phase 3 で確立済だったため、Phase 4 のスキーマ実装が高速

## 4. レビュー — 学び

- **CRITICAL ゼロ** ✓
- **HIGH 警戒点**:
  - release-composer 出力の `targets` が ChangeEntry 全体を返してしまった (string[] のみのはず)
  - → スキーマ検証時に entity 名のみに整形して merge することで対処
  - subagent system prompt に「targets は string array だけ」と書いていなかった → 改善余地
- **MEDIUM**:
  - 完走時間 130 秒で目標 90 秒未達 (Phase 7 で改善)

## 5. 修正・再実装 — 学び

- 1 度のリント修正 (`"x" + var` → template literal) のみ
- subagent 出力の整形は呼び出し側で柔軟に対応できる構造になっていた

## 6. 整理 — 学び
