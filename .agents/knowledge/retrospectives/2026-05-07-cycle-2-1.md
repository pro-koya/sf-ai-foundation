# Retrospective: Phase 2 サイクル 2-1

## サイクル要約

Phase 1 引き継ぎ improvement 6 項目のうち 5 項目をコード対応で解消。残る 1 項目 (`npm test` 実環境確認) は実行権限がない環境のため次サイクル以降に持ち越し。
新規モジュール 11 個 (`extractors/*` 8 + `sqlite-reader.ts` + `secrets/load.ts` + `extractors/types.ts`)、新規テスト 7 ケース、`as any` 完全排除、tsconfig 調整 1 件。

## 1. 要件整理 — 学び

- improvements/ + pitfalls/ から課題項目を順序付けして処理する運用が機能した
- 先行の改善 ADR (2026-05-07-phase-1-handover-improvements.md) を「やることリスト」として直接消化できた

## 2. 計画立案 — 学び

- 7 サブサイクル (A〜G) に分けたが、A (tsconfig 調整) は 1 行修正なので独立サイクル化は過剰だった。次回は前提条件をまとめて 1 サブにする
- B (extractors 分割) と E (apiVersion 修正) は密接に関連したので、E を B の後に配置した順序は正しかった

## 3. 実行 — 学び

- `Edit` で書こうとして「Read 先行」エラーに何度か遭遇 → 既存ファイル変更時はまず Read してから Edit/Write する習慣を再確認
- `KnowledgeGraphReader` を private クラスフィールド (`#store`) で書いたが、生成されるバンドルが ES2022+ ターゲットなので問題なし
- `extractors/types.ts` で `DEFAULT_API_VERSION` を共有定数化したことで、デフォルト値の単一情報源 (Single Source of Truth) を作れた

## 4. レビュー — 学び

- **CRITICAL ゼロ** ✓
- **HIGH チェック**:
  - `as any` 全削除 ✓
  - `KnowledgeGraphReader` の row interface はすべて explicit な型 ✓
  - 古い `extractors.ts` を削除し import 元を更新 ✓
- **MEDIUM 警戒点**:
  - `extractors/index.ts` から re-export している項目数が多い → ファイル直 import への切替を Phase 5 で検討
  - `process.cwd()` を `secrets/load.ts` で参照している → デフォルト引数なので問題なし

## 5. 修正・再実装 — 学び

- 大きな手戻りなし
- ただし `apex-class.ts` で `void dirname` という残骸を一度書いてしまい即削除。生成 AI 特有の「とりあえず import」癖に注意

## 6. 整理 — 学び

- 古いファイル (`extractors.ts` 単一) を `rm` で削除し、import 先を Edit で更新する流れがクリーン
- `src/index.ts` を都度更新したことで public API の追加忘れを防げた

## 良かった点 (Keep)

- **improvements 起点で対応する運用**: 「やることリスト」が ADR として残っていたので迷いがなかった
