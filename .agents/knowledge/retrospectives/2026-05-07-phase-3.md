# Retrospective: Phase 3 (3-1〜3-7)

## サイクル要約

差分意味づけと再現性ガバナンス本格稼働の Phase 3 を、検証ゲート (3-8) を除いて構造的に完成。

- **3-1**: `yohaku diff` CLI (git numstat 連携、メタデータ型 12 種を拡張子で決定的判定、7 + unknown カテゴリ)
- **3-2**: change_summary スキーマ (源泉 source 列必須、Tracked&lt;T&gt; / ai / human / deterministic ヘルパー、ajv 強制)
- **3-3**: 5 並列分類 subagent (data-model / automation / permission / ui / logic)
- **3-4**: `/classify-diff` slash command (Task 並列起動 + 統合)
- **3-5**: 一致率 CI 基盤 (`runConsistencyCheck` + `expectMatchRate`)
- **3-6**: SARIF 取り込み (`yohaku diff --include-static-analysis`)
- **3-7**: `/change-summary` slash command (PR Markdown サマリ + 大型差分サンプリング指針)

CI: 82/82 pass、実環境 build 成功。

## 1. 要件整理 — 学び

- 利用者の検証ゲート (Phase 2-8) で symlink + FK の 2 件のバグを抽出した経験から、Phase 3 では「実 AI 呼び出しを伴わない構造的検証」を最大化した
- 一致率 CI は producer 抽象化により、実 AI 呼び出しなしで枠組みを単体テスト可能にできた

## 2. 計画立案 — 学び

- Phase 3 計画 ADR で 8 サイクル分割を事前に明確化したのが効いた → 各サイクルの責務が独立、依存も線形
- 3-2 (Tracked&lt;T&gt; + schema) を 3-3 (subagent) より前に置く順序は正解 — subagent の system prompt が schema を参照する関係

## 3. 実行 — 学び

- `Tracked<T>` の wrapper 構造は冗長だが、JSON Schema レベルで source を強制することの価値は計り知れない (ajv テストで未指定が即 fail)
- `runConsistencyCheck` の `canonicalize` ロジックを sort-key JSON にしたことで、object key 順序によらない比較が可能に
- SARIF パーサは「最小サブセット」アプローチが正解 — 完全な仕様準拠ではなく Code Analyzer v5 が出力する範囲のみ

## 4. レビュー — 学び

- **CRITICAL ゼロ** ✓
- **HIGH 警戒点**:
  - `runConsistencyCheck` の producer に実 AI を差し込むのは Phase 5 — 現状はモック前提のテスト
  - ゴールデンケース 1 件のみ → Phase 5 で 3 件以上に拡充
- **MEDIUM**:
  - `/classify-diff` の subagent への入力 (各カテゴリのファイルリスト) を Claude Code Task ツールがどう受け取るか実機検証必要 (3-8)

## 5. 修正・再実装 — 学び

- 大きな手戻りなし
- biome auto-format が 5 ファイル整形 (主に test ファイル) — 既存パターンと整合

## 良かった点 (Keep)

- **Tracked&lt;T&gt; パターン**: source 列の wrapper を ajv で強制する設計、再現性ガバナンスの中核として機能
- **producer 抽象化**: 一致率 CI を実 AI 無しでテスト可能に
- **SARIF 最小サブセット**: 完全準拠を狙わず、Code Analyzer v5 出力に絞った判断
- **scaffold subagent の責務分離**: 5 種が「これは扱わない」を system prompt に明記、相互侵食を防止
- **メタデータ型 12 種を拡張子で判定**: Phase 1 既知制約 (LWC/Aura 等) を Phase 3 で前倒し対応した部分もある
- **段階的サイクル**: 3-1 → 3-2 → 3-3 → ... の依存順を計画 ADR で明確化、手戻りゼロ

## 課題 (Problem)

- 一致率 CI が実 AI 統合まで進めなかった (Phase 5 へ繰越)
- ゴールデンケース 1 件のみ (バリエーション不足)
- `/classify-diff` の Task 並列起動の **コスト・レイテンシ** は実機検証 (3-8) でのみ判明

## 試したいこと (Try)

- Phase 3-8 で利用者が `/classify-diff` を実 PR に対して実行 → AI コストとレイテンシを記録
- Phase 5 で実 Claude API を `runConsistencyCheck` に差し込み、3 ケースで N=5 一致率を測る
