# Retrospective: Phase 5

## サイクル要約

オンボーディング本格化の 6 サイクルを yohaku-trial 実プロジェクトで完走。検証ゲート通過。

- 5-1: context-map ローダ (YAML, 3 段フォールバック) + 4 persona デフォルト
- 5-2: onboarding-state.json store (recordStep / increment / reset)
- 5-3: persona 別 subagent 3 種 (review-assistant / release-advisor / customer-impact-explainer)
- 5-4: `yohaku onboard {context, state, faq}` CLI + `/onboard --role` 全面刷新
- 5-5: FAQ 抽出 + PII フィルタ (secrets/mask 再利用)
- **5-6: yohaku-trial で 4 機構の実機動作確認 + review-assistant 起動でレビュー観点 5 件抽出**

CI 101/101 pass。

## 1. 要件整理 — 学び

- IMPLEMENTATION_GUIDE.md Phase 5 ゴール (4 persona + context-map + state + FAQ) を **6 サイクルでほぼ完全カバー**
- 利用者の yohaku-trial に **既に蓄積された change_summary / release_doc** が、Phase 5 の subagent 群の入力として機能した — Phase 3〜4 の成果が Phase 5 で結実

## 2. 計画立案 — 学び

- 5-3 (subagent) より先に 5-1 / 5-2 (基盤) を作る順序が正解 — subagent system prompt で `yohaku onboard context` を呼べる前提が立った
- FAQ 抽出は secrets/mask の **再利用** で実装コスト最小化 (Phase 1 で作った基盤が効いた)

## 3. 実行 — 学び

- `splitSections` のロジックを最初は `\n# 質問\n` のような単純 split で書いて失敗 → 行ベースの state machine に書き直して通った
- `domains:*` という特殊トークンを expandReadOrder で展開する設計は、context-map.yaml の可読性を保ちつつ柔軟
- biome auto-format で 3 ファイル整形 + 1 件 useTemplate 修正 (既知パターン)

## 4. レビュー — 学び

- **CRITICAL ゼロ** ✓

  - FAQ 抽出は単純 Markdown 形式のみサポート (Claude Code session JSON 形式は Phase 7)
